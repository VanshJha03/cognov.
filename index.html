Introduction

CognoV provides enterprise ledger accounts as an API utility. Developers integrate CognoV directly into their e-commerce checkouts, billing systems, or ERP systems to convert business operations into double-entry accounting records automatically.

Your inventory management app communicates directly with our API endpoints. You do not route calls through this frontend dashboard; your app calls your Cloudflare worker backend directly.

How Output is Received

CognoV utilizes a synchronous Request-Response architectural lifecycle. When your inventory management system issues a transaction request, the execution flow behaves as follows:

  1. HTTP POST request: Your app sends payload data containing the transaction type, quantities, and pricing to the /transaction endpoint.
  2. Synchronous Execution: The backend calculates unit cost tracking, posts debit/credit entries, updates the trial balances/balance sheets, and records database transactions.
  3. HTTP JSON Response: The backend immediately returns the ledger outcome in the HTTP response body. Your inventory system parses this JSON directly from the connection response stream.

Example Request-Response Body Mappings

1. Sales Event Response Schema

Returned immediately upon a successful sales posting, containing computed cash changes and Cost of Goods Sold (COGS) metrics.

{
  "debit": "Cash",
  "credit": "Sales Revenue",
  "revenue": 950.00,
  "cogs": 500.00
}
2. Purchase/Inventory Inflow Response Schema

Returned immediately upon a stock acquisition purchase posting, specifying asset classification changes and totals.

{
  "debit": "Inventory",
  "credit": "Cash",
  "amount": 250.00
}

Response Status Codes

  • HTTP 200 OK: Transaction processed successfully. Ledger records logged.
  • HTTP 400 Bad Request: Insufficient inventory, unknown item, or invalid payload schema.
  • HTTP 401 Unauthorized: The Developer API Key is missing or invalid.
  • HTTP 429 Too Many Requests: Client rate limits (100req/min) exceeded.

Authentication

All API requests must supply an authentication header or URL query parameter. There are two tiers of credentials:

Owner Key
String
Superkey representing the main entity. Carries master permissions to run ledger queries for any client under its node. Format: cognov_xxxxxxxxxx.
api_key
Query Param
Provide authorization via query param: ?api_key=your_owner_key.

Register User Node

POST /register_user

Creates a new tenant user node under the owner account. Generates an isolated ledger structure.

Query Parameters

api_key
required
Owner master key.

Response Payload

{
  "user_id": "a1b2c3d4",
  "api_key": "X9z2LpQ4R1"
}

Execute Transaction

POST /transaction

Submits business inventory or cash movements to compile debit/credit journals.

Request Body schema

api_key
string
Owner master key credentials.
user_id
string
The specific client target.
transaction
object
Payload detailing the action. Schema varies by event type (sales, purchase, initialize, write_off, etc).

Examples by Action Type

// Sales Action
{
  "api_key": "X9z2LpQ4R1",
  "user_id": "a1b2c3d4",
  "transaction": {
    "action": "sales",
    "item": "Keyboard",
    "qty": 3,
    "selling_price": 45
  }
}

Fetch Accounting Reports

GET /fetch/{user_id}/{report_type}

Retrieves structured accounting statements compiled from transaction history.

Path Parameters

user_id
required
8-digit ID of the targeted client node.
report_type
required
Specify report: journal, trialbalance, or balancesheet.

🔒 Privacy Compliance Policy

Effective Date: June 14, 2026

1. Scope of Data Ingestion

CognoV processes transaction metadata (identifiers, quantities, values, event timestamps) and compiled double-entry journal entries solely to execute balances and reports. We do not inspect personal billing accounts or payment card identifiers.

2. Data Processing & Isolation

All client nodes run in isolated database schemas. Transaction values, quantities, and inventory reports generated through the API are private to the registered client. Owner key accounts carry control credentials over their respective nodes.

3. Data Security & Storage

Database instances are stored with encryption-at-rest (using AES-256) and data transfers occur encrypted in transit (TLS 1.3). API tokens are handled using cryptographic hashing layers before DB records generation.

4. Retention and Deletion

Transaction records are retained for compliance audit trails. Database nodes can be completely purged or deleted at any time through the administration API. To initiate an account termination, contact security@cognov.com.
